# 6.0 — Transformer: Duas Arquiteturas para Gap-Filling Climático

Contexto: 47M amostras, 79 features (k=15 vizinhos), 5 variáveis, MinMax [0,1].  
Mesmo `train_scaled.parquet` / `test_scaled.parquet` usados em 4.0 e 5.0.

---

## Comparação Rápida

| | **Arquitetura A — FT-Transformer** | **Arquitetura B — SNT (Spatial Neighbor Transformer)** |
|---|---|---|
| Tokens | 80 (1 por feature + CLS) | 16 (1 por vizinho + 1 temporal) |
| Atenção | 80 × 80 = 6.400 scores | 16 × 16 = 256 scores |
| Física do dado | Abstrata (features independentes) | **Natural** (token = estação física) |
| Velocidade | ~50s / variável | ~20s / variável |
| Interpretabilidade | Importância por feature | Qual vizinho o modelo priorizou |
| Parâmetros (wide) | ~3.5M | ~1.2M |
| Risco | Overfit em features redundantes | Depende do reshape correto |
| **Recomendação** | Segundo teste | **Primeiro teste** |

## O que é um Transformer (resumo técnico)

```
Input → Tokenizar → [Self-Attention → FFN] × N camadas → Output agregado → MLP → Predição

Self-Attention (o núcleo):
  Para cada token, calcula quanto ele deve "prestar atenção" a cada outro token.
  
  Q = X · W_q   ← "o que estou procurando?"
  K = X · W_k   ← "o que sou?"
  V = X · W_v   ← "qual informação carregar?"

  Attention = softmax(Q · K^T / √d_k) · V
            ↑
            Matriz 16×16 ou 80×80 de pesos de atenção.
            O modelo aprende quais tokens importam para cada predição.

Multi-head: repete atenção em H cabeças independentes, cada uma
            aprende um padrão diferente (ex: cabeça 1 = distância,
            cabeça 2 = azimute, cabeça 3 = altitude, ...)
```

## Arquitetura A — FT-Transformer (Feature Tokenizer + Transformer)

**Referência:** Gorishniy et al., NeurIPS 2021 — "Revisiting Deep Learning Models for Tabular Data"

### Conceito

Cada uma das 79 features vira um token. O Transformer aprende quais features
interagem entre si — ex: "quando n01 é alto E d01 é baixo E hora é meio-dia → temperatura alta."

```
Input (batch, 79)
    │
    ├── feature_1 → Linear(1 → D) ─────────┐
    ├── feature_2 → Linear(1 → D) ─────────┤
    │   ...                                 │ 79 tokens
    └── feature_79 → Linear(1 → D) ────────┤
                                            │
[CLS token] (parâmetro aprendido) ──────────┘
                                            │
                                       (batch, 80, D)
                                            │
                        ┌───────────────────┘
                        │
               ┌────────▼────────┐
               │  Multi-Head     │
               │  Self-Attention │  ← 80×80 = 6.400 scores
               │  (H cabeças)    │
               └────────┬────────┘
                        │ Residual + LayerNorm (Pre-Norm)
               ┌────────▼────────┐
               │  FFN: D → 4D → D│  ← GELU, 2 camadas
               └────────┬────────┘
                        │ Residual + LayerNorm
                        │
               Repete N vezes (N=3, 6, ou 12)
                        │
                 CLS output (batch, D)
                        │
               ┌────────▼────────┐
               │  MLP Head       │  D → 256 → 1
               │  GELU, Dropout  │
               └────────┬────────┘
                        │
                  Predição (batch,)
```

## Arquitetura B — SNT (Spatial Neighbor Transformer)

### Conceito

**Cada estação vizinha é um token.** Em vez de 79 features abstratas, a entrada
é reorganizada em 15 tokens geoespaciais + 1 token temporal.

```
Features originais (79):
  n01, d01, a01, b01_sin, b01_cos  ←→  vizinho 1 (5 features)
  n02, d02, a02, b02_sin, b02_cos  ←→  vizinho 2 (5 features)
  ...
  n15, d15, a15, b15_sin, b15_cos  ←→  vizinho 15 (5 features)
  hour_sin, hour_cos, doy_sin, doy_cos  ←→  contexto temporal (4 features)

Reshape → 16 tokens × 5 features (padding=0 no temporal)
```

```
Input (batch, 79)
    │
    ├── Reshape em 16 grupos × 5 features
    │
    ├── vizinho_1  [n01, d01, a01, b01_sin, b01_cos] ──→ Linear(5 → D) + PE_spatial
    ├── vizinho_2  [n02, d02, a02, b02_sin, b02_cos] ──→ Linear(5 → D) + PE_spatial
    │   ...                                                    15 tokens
    ├── vizinho_15 [n15, d15, a15, b15_sin, b15_cos] ──→ Linear(5 → D) + PE_spatial
    │
    └── temporal   [hour_sin, hour_cos, doy_sin, doy_cos, 0] ─→ Linear(5 → D)
                                                               1 token
                                              (batch, 16, D)
                                                    │
                           ┌────────────────────────┘
                           │
                  ┌────────▼────────┐
                  │  Multi-Head     │
                  │  Self-Attention │  ← 16×16 = 256 scores (25× menor que FT-T)
                  │  (H cabeças)    │
                  └────────┬────────┘
                           │ Residual + Pre-LayerNorm
                  ┌────────▼────────┐
                  │  FFN: D → 4D → D│
                  └────────┬────────┘
                           │
                  Repete N vezes
                           │
               Mean pooling sobre 16 tokens  ← ou usar o token temporal como CLS
                           │
                  ┌────────▼────────┐
                  │  MLP Head       │  D → 256 → 1
                  └────────┬────────┘
                           │
                     Predição (batch,)
```

### Position Encoding Espacial (PE_spatial)

O PE_spatial injeta a posição geoespacial de cada vizinho no espaço de embedding.
As features `d_i` (distância) e `b_i_sin/cos` (azimute) já estão no vetor do token,
mas o PE adiciona um encoding posicional explícito baseado nelas — similar a como
BERT usa encoding de posição absoluta, mas aqui a "posição" é geoespacial.

```python
def spatial_positional_encoding(d_normalized, b_sin, b_cos, dim):
    """
    d_normalized: distância [0,1]
    b_sin, b_cos: azimute em sin/cos
    dim: embedding_dim

    Resultado: vetor PE ∈ ℝ^dim adicionado ao token embedding.
    """
    k = torch.arange(dim // 4)

    # Fourier para distância
    pe_d  = [sin(d / 10000**(2k/dim)) para k=0..dim/4]
            + [cos(d / 10000**(2k/dim)) para k=0..dim/4]

    # Encoding de azimute (já em sin/cos, usar diretamente)
    pe_az = [b_sin * sin(k·π/dim) para k=0..dim/4]
            + [b_cos * cos(k·π/dim) para k=0..dim/4]

    return pe_d + pe_az  # ∈ ℝ^dim

# Intuição:
# Vizinho a 10km ao norte → PE específico
# Vizinho a 200km ao sul  → PE bem diferente
# O modelo aprende: tokens com PE de "perto" merecem mais atenção
```

### O que a atenção aprende

Após treino, cada cabeça de atenção aprende um padrão geoespacial diferente:

```
Exemplo hipotético após treino em temperatura:

Cabeça 1 — "Vizinhança imediata":
  vizinho_1 (12km, nordeste):  atenção = 0.28  ← alto
  vizinho_2 (18km, norte):     atenção = 0.22  ← alto
  vizinho_8 (185km, sul):      atenção = 0.03  ← baixo
  ...

Cabeça 2 — "Mesmo vale / mesma altitude":
  vizinho_3 (45km, delta_alt=-20m): atenção = 0.31  ← alto, mesma altitude
  vizinho_9 (90km, delta_alt=-15m): atenção = 0.19
  vizinho_5 (30km, delta_alt=+400m): atenção = 0.04  ← baixo, altitude diferente

Cabeça 3 — "Sazonalidade (interage com token temporal)":
  token_temporal: atenção = 0.45  ← alto — o momento do dia importa muito
  vizinho_1: atenção = 0.18
  ...
```

### Input reshape (implementação)

```python
def build_neighbor_tokens(X: torch.Tensor, k: int = 15) -> torch.Tensor:
    """
    X: (batch, 79) — features na ordem padrão do pipeline
    
    Retorna tokens: (batch, 16, 5)
      tokens[:, 0:15, :] = 15 vizinhos, cada um com [n, d, a, b_sin, b_cos]
      tokens[:, 15, :]   = contexto temporal [hour_sin, hour_cos, doy_sin, doy_cos, 0]
    """
    B = X.shape[0]
    tokens = torch.zeros(B, k + 1, 5, device=X.device, dtype=X.dtype)

    for i in range(k):
        tokens[:, i, 0] = X[:, i]           # n{i+1}  — medição
        tokens[:, i, 1] = X[:, k + i]       # d{i+1}  — distância
        tokens[:, i, 2] = X[:, 2*k + i]     # a{i+1}  — delta altitude
        tokens[:, i, 3] = X[:, 3*k + i]     # b{i+1}_sin
        tokens[:, i, 4] = X[:, 4*k + i]     # b{i+1}_cos

    # Token temporal (últimas 4 features do pipeline)
    tokens[:, k, 0] = X[:, 5*k]     # hour_sin
    tokens[:, k, 1] = X[:, 5*k + 1] # hour_cos
    tokens[:, k, 2] = X[:, 5*k + 2] # doy_sin
    tokens[:, k, 3] = X[:, 5*k + 3] # doy_cos
    # tokens[:, k, 4] = 0.0          # padding

    return tokens  # (batch, 16, 5)
```

### Vantagens

- **Fisicamente motivado**: cada token = estação física real
- **Eficiente**: 16 tokens → atenção 16×16 = 25× menor que FT-T 80×80
- **Interpretável**: os pesos de atenção revelam diretamente quais vizinhos importaram
- **Generaliza melhor**: tokens com estrutura (5 features geoespaciais) aprendem padrões mais ricos que tokens de 1 feature
- **Treino mais rápido**: batch maior, sequência menor → ~20s/variável

### Desvantagens

- Requer o reshape correto dos 79 features (implementação crítica)
- Depende da ordem das colunas do pipeline (n01..n15, d01..d15, etc.)
- Não captura interações ENTRE features do mesmo vizinho dentro do token
  (ex: "medição alta E distância perto" precisa ser aprendida no Linear embedding)


## Sugestão de Combinações Adicionais

Qualquer uma das duas arquiteturas pode ser combinada com:

| Técnica | Como | Ganho esperado |
|---------|------|----------------|
| **Ensemble Transformer + MLP** | média ponderada `0.6×T + 0.4×MLP` | +3-5% |
| **Ensemble Transformer + RF** | `0.5×T + 0.3×MLP + 0.2×RF` | +2-4% adicional |
| **Flash Attention** | substitui `nn.MultiheadAttention` por `flash_attn` | sem ganho em MAE, 2-3× mais rápido |
| **Gradient clipping** | `clip_grad_norm_(model, 1.0)` | estabilidade de treino |
| **Warmup de LR** | `LinearWarmup(steps=1000)` antes do scheduler | convergência mais suave |

---

## Recomendação

**Comece pela Arquitetura B (SNT).**

Motivos:
1. Os 79 features do pipeline têm estrutura natural de "15 vizinhos × 5 atributos" — SNT aproveita isso diretamente.
2. Treino ~2.5× mais rápido que FT-T (sequência de 16 vs 80 tokens).
3. Atenção sobre vizinhos é interpretável: após treino, dá para plotar "qual estação física o modelo priorizou" para cada predição.
4. Ganho estimado ligeiramente superior ao FT-T por capturar melhor a estrutura espacial do problema.

Se SNT não superar o MLP, ou para validação, testar FT-T em seguida.
