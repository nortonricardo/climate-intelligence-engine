# 1.5 — Build Neighbors

## O que é e para que serve

O script `1.5_build_neighbors.py` enriquece cada medição meteorológica com informações das estações vizinhas mais próximas disponíveis no mesmo instante de tempo. O resultado é um dataset pronto para treinar modelos de **gap-filling** — técnica que estima o valor de uma estação quando ela não possui medição em determinado timestamp.

Para cada variável (`temperature`, `humidity`, `rainfall`, `global_radiation`, `pressure`), é gerado um arquivo Parquet independente onde cada linha representa uma medição real de uma estação-alvo, acompanhada das medições e características espaciais das 20 estações mais próximas que também possuíam dado naquele mesmo momento.

---

## Contexto: por que precisamos dos vizinhos?

Estações meteorológicas frequentemente apresentam falhas — sensor com defeito, queda de energia, manutenção. O objetivo do projeto é estimar esses valores ausentes com precisão. A hipótese central é:

> **Estações próximas tendem a medir valores semelhantes no mesmo instante, com diferenças previsíveis em função da distância, altitude e direção.**

Portanto, se a estação A001 não tem temperatura às 14h de uma terça-feira, mas 15 estações ao seu redor têm, podemos estimar o valor de A001 a partir dessas medições vizinhas — pesando cada vizinho de acordo com sua proximidade, diferença de altitude e direção.

---

## Algoritmo

### 1. Pivot da variável

O arquivo de entrada (`temperature.parquet`, etc.) tem o formato longo:

```
code | time | measurement
A001 | 2020-01-01 12:00 | 22.5
A002 | 2020-01-01 12:00 | 23.1
...
```

O script transforma isso em uma matriz **tempo × estação**:

```
         A001  A002  A003  ...
12:00    22.5  23.1   NaN
13:00    23.0   NaN  24.2
...
```

Células `NaN` indicam que a estação não possui dado naquele timestamp. Essa matriz permite consultar todos os vizinhos de uma estação em todos os seus timestamps com uma única operação vetorial.

### 2. Índice de distâncias

O arquivo `station_distances.parquet` (gerado pelo `1.2`) contém as distâncias pré-calculadas entre todos os pares de estações, ordenadas por `effective_distance_km`. O índice é carregado em memória como um dicionário:

```
from_code → (to_codes[], distance_km[], delta_altitude_m[], bearing_sin[], bearing_cos[])
```

O cálculo do azimute (bearing) é feito neste momento, usando as coordenadas de `stations.parquet`.

### 3. Seleção dos K primeiros vizinhos válidos

Para cada estação-alvo, o algoritmo:

1. Filtra a matriz pivotada para as linhas onde a estação-alvo tem dado
2. Extrai a sub-matriz de vizinhos (linhas com dado × colunas dos vizinhos ordenados por distância)
3. Aplica `first_k_valid_multi`: varre a máscara de NaN uma única vez e seleciona os **20 primeiros vizinhos não-nulos** em cada linha simultaneamente para valores, distâncias, altitudes, azimutes

O resultado são cinco matrizes `(n_timestamps, 20)` — uma por feature espacial — com `NaN` nos slots onde não havia 20 vizinhos disponíveis.

### 4. Escrita incremental

Os dados são escritos estação por estação via `PyArrow ParquetWriter`, sem acumular tudo em memória. Isso permite processar o dataset completo (~70M registros por variável) em uma máquina comum.

## Schema de saída

Cada arquivo possui **107 colunas**:

### Identificação (2)

| Coluna | Tipo | Descrição |
|---|---|---|
| `code` | string | Código da estação-alvo (ex: `A001`) |
| `time` | timestamp | Data e hora da medição (UTC) |

### Medição da estação-alvo (1)

| Coluna | Tipo | Descrição |
|---|---|---|
| `measurement` | float32 | Valor medido pela estação-alvo |

> Para `temperature`, o valor é arredondado para 2 casas decimais.

### Medições dos vizinhos — `n01` a `n20` (20)

Valor medido por cada vizinho disponível no mesmo timestamp, ordenado do mais próximo para o mais distante (por `effective_distance_km`). `NaN` indica que menos de K vizinhos estavam disponíveis naquele instante.

### Distância horizontal — `d01` a `d20` (20)

Distância geodésica (Haversine) em quilômetros entre a estação-alvo e o vizinho que ocupa aquele slot. Varia por linha porque o vizinho no slot k muda conforme a disponibilidade de dados em cada timestamp.

**Fórmula de Haversine:**
```
a = sin²(Δlat/2) + cos(lat1)·cos(lat2)·sin²(Δlon/2)
d = 2R · arcsin(√a)        R = 6371 km
```

### Diferença de altitude — `a01` a `a20` (20)

Diferença de altitude em metros entre o vizinho e a estação-alvo (`altitude_vizinho - altitude_alvo`). Valor positivo significa que o vizinho está mais alto; negativo, mais baixo.

Altitude influencia diretamente temperatura (lapse rate: ~6.5°C/1000m), pressão atmosférica e regime de chuvas.

### Azimute — `b01_sin` / `b01_cos` a `b20_sin` / `b20_cos` (40)

Direção do vizinho em relação ao alvo, codificada em seno e cosseno para preservar a continuidade circular (0° e 360° são a mesma direção, mas numericamente distantes).

**Fórmula do azimute (forward bearing):**
```
Δlon = lon_vizinho - lon_alvo
x    = sin(Δlon) · cos(lat_vizinho)
y    = cos(lat_alvo) · sin(lat_vizinho) − sin(lat_alvo) · cos(lat_vizinho) · cos(Δlon)
b    = atan2(x, y)          ∈ [−π, π] radianos

b_sin = sin(b)
b_cos = cos(b)
```

**Por que seno e cosseno em vez do ângulo em graus?**

Um modelo que recebe o ângulo como número puro não sabe que 1° e 359° são praticamente a mesma direção (apenas 2° de diferença). Com `sin`/`cos`, essa continuidade é preservada automaticamente — o mesmo princípio dos encodings temporais cíclicos.

**Por que o azimute importa?**

O clima tem direcionalidade. Uma frente fria que avança do sul faz com que estações ao sul do alvo sejam mais preditivas do que estações ao norte à mesma distância. Os alísios de leste tornam estações a leste mais relevantes para umidade e precipitação. Sem o azimute, o modelo só sabe *quão longe* está cada vizinho, não *de onde* ele está.

### Features temporais cíclicas (4)

| Coluna | Fórmula | Representa |
|---|---|---|
| `hour_sin` | `sin(2π · hora / 24)` | Posição no ciclo diário |
| `hour_cos` | `cos(2π · hora / 24)` | Posição no ciclo diário |
| `doy_sin` | `sin(2π · dia_do_ano / 365)` | Posição no ciclo sazonal |
| `doy_cos` | `cos(2π · dia_do_ano / 365)` | Posição no ciclo sazonal |

**Por que encoding cíclico?**

Temperatura e radiação têm ciclos diários e sazonais fortíssimos. Sem informação temporal, o modelo não distingue entre uma medição de 3h da manhã e outra das 15h. Com a hora como número puro (0–23), o modelo veria 23 e 0 como extremos opostos — mas 23h e 0h são apenas 1 hora de diferença. O encoding `sin`/`cos` resolve isso: a distância matemática entre dois horários reflete sua proximidade real no ciclo.

## Decisões de design

### Por que d e a separados e não effective_distance?

`effective_distance_km = sqrt(d² + (a/100)²)` é uma métrica composta que combina distância horizontal e altitude em um único número. Usá-la isoladamente descarta a informação individual de cada componente.

Modelos de ML (Random Forest, MLP, Transformer) aprendem pesos diferentes para distância horizontal e diferença de altitude dependendo da variável: pressão é muito mais sensível à altitude do que à distância; precipitação pode depender mais da orografia (relevo) do que da distância plana. Tendo `d` e `a` separados, o modelo aprende esses pesos — com `effective_distance` apenas, não consegue.

### Por que não armazenar os pesos IDW?

IDW (Inverse Distance Weighting) é uma família de métodos onde `w_i = 1/d_i^p`. Armazenar `w_i` pré-calculado (com `p=2`, por exemplo) travaria a escolha do expoente no momento da geração dos dados. Além disso, modelos não-lineares como RF e Transformer aprendem seus próprios pesos implicitamente — o peso IDW pré-calculado seria uma transformação que remove informação (não é possível recuperar `d` a partir de `w`).

### Por que 20 vizinhos?

Compromisso entre cobertura e tamanho do arquivo. Com 663 estações e dados de 2000–2025, a maioria dos timestamps tem entre 300 e 450 estações com dado. 20 vizinhos capturam o contexto espacial imediato sem tornar o schema excessivamente largo para modelos simples.

## Inputs e outputs

**Inputs:**
- `data/{variável}.parquet` — gerado pelo `1.4_clean_data.py`
- `data/station_distances.parquet` — gerado pelo `1.2_compute_station_distances.py`
- `data/stations.parquet` — gerado pelo `1.1_download_data.py`

**Outputs** (um por variável):
- `data/temperature_neighbors.parquet`
- `data/humidity_neighbors.parquet`
- `data/rainfall_neighbors.parquet`
- `data/global_radiation_neighbors.parquet`
- `data/pressure_neighbors.parquet`

> Com filtro de data ativo (`DATE_START` / `DATE_END`), o sufixo de datas é adicionado ao nome do arquivo — ex: `temperature_neighbors_2025-03-25_2025-03-25.parquet`.
